# 집콕신혼 — 통합 디자인 브리프

> **서비스**: 수도권 신혼부부 주거 분석 서비스 (정보 분석 플랫폼)
> **기술 스택**: React + Next.js App Router · Tailwind CSS v4 · Recharts · 네이버 지도 API
> **최종 작성일**: 2026-02-13
> **대상 독자**: 개발팀 (프론트엔드·백엔드·QA)

---

## 1) 한 줄 요약

**맞벌이 신혼부부의 예산·이중통근·육아·안전을 한 번에 분석하여, 커플이 비동기로 함께 의사결정할 수 있는 공공데이터 기반 주거 정보 플랫폼.** [source: P1, P2]

## 2) 디자인 원칙

P2에서 도출된 10개 원칙 중 핵심 7개를 요약한다.

| # | 원칙 | 한 줄 요약 | 핵심 구현 규칙 | source |
|---|------|----------|--------------|--------|
| 1 | **비동기 협업 우선** | 커플의 탐색은 각자 따로, 짧게, 반복적으로 이루어진다 | 공유 찜 목록 + "둘 다 찜/한 명만 찜" 매칭 표시를 기본 제공한다 | P2 §1 |
| 2 | **3입력 내 첫 가치 전달** | 소득·자산·대출 3개 입력만으로 예산 범위를 즉시 보여준다 | 온보딩 완료 후 홈이 아닌 결과 화면(`/results`)으로 직접 라우팅한다 | P2 §2 |
| 3 | **지도가 곧 검색이다** | 바운딩 박스 자동 갱신 + 가격 라벨 핀 + 3단 바텀시트 | 지도 팬/줌 시 별도 "검색" 버튼 없이 자동 재검색한다 | P2 §3 |
| 4 | **점수는 숫자가 아니라 이야기** | 0-100 점수 + 라벨 + 요인 분해 + 백분위 비교 4층 구조 | 모든 점수에 ⓘ → 1줄 요약 → 요인 기여도 → 전체 방법론 3단계 계층적 공개를 구현한다 | P2 §4 |
| 5 | **실거래가가 최고의 신뢰 장치** | 호가 옆에 동일 조건 실거래 이력을 항상 병기 | 모든 가격 데이터에 국토교통부 출처·기준일을 인라인 태그로 표시한다 | P2 §5 |
| 6 | **점진적 몰입** | 등록 없이 탐색 → 카카오 로그인으로 찜 → 파트너 초대 → 외부 사이트 | 결과 확인까지 로그인을 요구하지 않는다 (찜/알림/공유 시점에 유도) | P2 §6 |
| 7 | **신뢰 신호는 적지만 깊게** | 과도한 배지 나열은 역효과 (전환율 12% 하락 사례) | 검증된 소수의 신뢰 장치(실거래가·출처 태그·🔒 비저장 메시지)를 콘텐츠에 유기적으로 통합한다 | P2 §5 |

## 4) 페이지 구조 요약

### 4-1. 사이트맵 (1단계 트리) [source: P7 §1]

```
🌐 / (홈·랜딩)
├── /search           조건 입력 (3스텝 위저드)
├── /results          분석 결과 (지도 + 바텀시트 리스트)
├── /complex/[id]     단지 상세
├── /compare          단지 비교 (최대 3개)
├── /board/[id]       공유 보드 (커플 협업)
├── /guide            신혼부부 주거 가이드
├── /auth             로그인·회원가입·콜백
├── /mypage           마이페이지 (저장·이력·알림·파트너·구독)
├── /terms            이용약관
├── /privacy          개인정보처리방침
└── /location-terms   위치정보 이용약관
```

### 4-2. 핵심 라우팅 테이블 [source: P7 §2]

| URL | 페이지 제목 | 인증 필요 | 렌더링 |
|-----|----------|----------|--------|
| `/` | 집콕신혼 — 수도권 신혼부부 주거 추천 | ❌ | SSG |
| `/search` | 조건 입력 | ❌ | CSR |
| `/results` | {지역명} {거래유형} 분석 결과 | ❌ | SSR |
| `/complex/[id]` | {단지명} — {동/구} {가격} | ❌ | ISR(1h) |
| `/compare` | {단지A} vs {단지B} 비교 | ❌ | CSR |
| `/board/[id]` | 우리의 후보 매물 | 쓰기 🔐 / 읽기 ❌ | SSR |
| `/guide/[slug]` | {가이드 제목} | ❌ | ISR(24h) |
| `/mypage/**` | 마이페이지 | 🔐 로그인 | CSR |
| `/mypage/alerts` | 가격 변동 알림 | 💎 프리미엄 | CSR |

### 4-3. 모바일 하단 탭 구성 (5탭) [source: P7 §3-1]

```
🏠 홈  |  🔍 검색  |  🗺️ 지도  |  📊 비교 ②  |  👤 MY
 /        /search   /results    /compare      /mypage
                    ?view=map   (뱃지: 비교 담긴 수)
```

- 높이: 56px + `env(safe-area-inset-bottom)`
- 활성 탭: `#0891B2` + Bold / 비활성: `#9CA3AF` + Regular
- `/complex/[id]`, `/results` 스크롤 다운 시 탭 바 숨김 → 스크롤 업 시 복귀

## 5) 핵심 화면별 설계

### 5-1. 온보딩 (조건 입력) [source: P4 JSX, P6 §2]

**5스텝, 총 ~2분 30초 목표**

| Step | 제목 | 내용 | 소요 시간 |
|------|------|------|----------|
| 1 | 우리 부부는요 | 주거 형태(매매/전세/월세) + 자녀 계획 | ~25초 |
| 2 | 출퇴근 정보 | 직장1 + 직장2 주소 | ~40초 |
| 3 | 소득과 자산 | 합산 연봉 + 보유 현금 | ~30초 |
| 4 | 부채와 예산 | 기존 대출 + 월 주거비 | ~30초 |
| 5 | 분석 중… | 결과 전환 애니메이션 (4단계 로딩) | ~7초 |

**설계 근거** (P4 주석):
- "원 퀘스천 퍼 스크린"이 이상적이나, 8개 입력을 8스텝으로 하면 진행감 저하 → 유사 맥락끼리 2개씩 묶어 5스텝
- Step 1을 가벼운 선택지로 시작 → "쉽다" 인상 → 완료 심리 작동
- 금융 정보(Step 3-4)를 후반에 배치 → 매몰비용 효과 활용

**입력 UI 패턴 선택 결과**

| 입력 항목 | 채택 패턴 | source |
|----------|----------|--------|
| 주거 형태 (매매/전세/월세) | 이모지+라벨 카드 선택 (3개) | P4 JSX |
| 자녀 계획 | 이모지+라벨 카드 선택 (3개) | P4 JSX |
| 직장 주소 | 텍스트 입력 (자동완성) | P4 JSX |
| 연봉·자산·대출·월비용 | 빠른 입력 버튼 (+1,000만/+5,000만/+1억) + 직접 입력 병행 | P4 JSX, P2 §2 |

**신뢰 확보 UI**

| 요소 | 위치 | 형태 | source |
|------|------|------|--------|
| 🔒 입력 정보 비저장 배지 (Full) | 금융 정보 입력 폼 상단 (Step 3) | 인라인 배너 (primaryLight 배경, 아이콘+타이틀+부연) | P4 JSX, P6 §3-1 |
| 🔒 축약형 리마인더 | 각 금융 입력 화면 하단 | "이 금액은 분석 후 즉시 삭제됩니다" 1줄 | P4 JSX, P6 §2-3 |
| ⓘ "왜 이 정보가 필요한지" 툴팁 | 각 입력 필드 우상단 | 탭 시 말풍선으로 1줄 해설 | P4 JSX |
| 필수/선택 동의 | 입력 시작 직전 인라인 | 필수(개인정보·위치) + 선택(정밀 분석) 시각적 분리 | P6 §2-2 |

**분석 중 애니메이션 시퀀스** (Step 5)

| 순서 | 아이콘 | 텍스트 | 지속시간 |
|------|-------|--------|---------|
| 1 | 🏘️ | 입력 조건 분석 중… | 1200ms |
| 2 | 🚇 | 통근 경로 계산 중… | 1400ms |
| 3 | 📊 | 예산 적합도 시뮬레이션 중… | 1600ms |

### 5-2. 결과 화면 [source: P5]

#### 리스트 뷰 — 카드 정보 우선순위

**1순위 (항상 노출)**

| 정보 | 위치 | 스타일 |
|------|------|--------|
| 순위 배지 (1, 2, 3…) | 좌상단 | 상위 3개 코랄 배경, 이하 회색 |
| 단지명 | 순위 배지 오른쪽 | 16px Bold(700) |
| 종합 점수 원형 게이지 | 우상단 | 64px SVG 원형 게이지 + 숫자 |

**2순위 (핵심 정보)**: 주소(12px) → 세대수·준공·면적(11px dot 구분) → 매매/전세 실거래가(13px Bold/SemiBold)

**3순위**: 예산·통근·보육·안전 점수 (2×2 그리드, 아이콘+라벨+등급색 숫자)

**4순위 (하단)**: 직장1·2 통근시간(12px) + "+ 비교" 버튼(아웃라인)

**정렬**: 종합 점수순(기본) / 예산 적합도순 / 통근 시간순 — 수평 칩, 선택 시 `#0891B2` 배경

**페이지네이션**: "더 보기" 버튼 채택 (무한 스크롤·페이지네이션 미채택) [source: P5 §2-4]

#### 지도 뷰 — 마커 디자인

**점수 라벨 핀** 사용 (가격이 아닌 종합 점수 표시) [source: P5 §3-2]

| 상태 | 배경 | 텍스트 | 크기 | 그림자 |
|------|------|--------|------|--------|
| Default | `#FFFFFF` | 점수 등급 컬러 | 1.0x (30–34px, 터치영역 44–48px) | `0 2px 8px rgba(0,0,0,0.12)` |
| Selected/Hovered | 점수 등급 컬러 | `#FFFFFF` | 1.15x | `0 4px 16px {color}40` |
| Visited | `#FFFFFF` (70% opacity) | 점수 등급 컬러 | 1.0x | `0 1px 4px rgba(0,0,0,0.08)` |

**지도-리스트 양방향 싱크**: 카드 hover → 마커 하이라이트, 마커 클릭 → 카드 자동 스크롤, 바텀시트 스와이프 → 마커 하이라이트

**모바일 3단 넌모달 바텀시트**: Peek(~25%, 100–160px) / Half(~50%, 카드 2–3개) / Expanded(~90%, 전체 리스트)

- 바텀시트가 올라와 있어도 지도 인터랙션 항상 가능 (넌모달)
- 그랩 핸들(40×4px) + X 닫기 버튼 동시 제공

#### 점수 시각화 — 최종 채택 조합 [source: P5 §4-2]

| 용도 | 시각화 | 크기 | 라이브러리 |
|------|--------|------|----------|
| 종합 점수 (1개) | **원형 게이지** (SVG) | 카드 64px / 상세 Hero 80–96px / 미니 36–48px | 순수 SVG |
| 카테고리별 점수 (4개) | **수평 바 차트** | 높이 8px(compact 6px), 전체 너비, 바 간격 14px | Tailwind CSS |
| 2–3개 단지 비교 | **레이더 차트** | 최소 200×200px, 4축 | Recharts |

> 레이더 차트는 비교 뷰 전용 (3개 이상 겹치면 가독성 급락) [source: P2 §4]

#### 비교 기능 UI [source: P5 §6]

- 모바일 최대 **3개**, 데스크톱 최대 **4개** 비교
- 나란히 비교 테이블: 행 단위 비교, 최고 점수 셀에 "더 좋음" 하이라이트 (`brand500/8%` 배경)
- 가격 항목에는 하이라이트 **미적용** (저가가 반드시 좋은 것은 아님)
- 하단 sticky 비교 바: 높이 64–80px, 배지 업데이트, `pb-[max(10px,env(safe-area-inset-bottom))]`
- 레이더 차트 단지 색상: `#0891B2` / `#F97316` / `#8B5CF6`, 채움 투명도 10%, 스트로크 2px

### 6-2. 동의 플로우 요약 [source: P6 §2]

```
랜딩 (가치 전달, 동의 없음)
  → 입력 시작 직전 (인라인 동의 수집)
    → 입력 (원퀘스천퍼스크린, 비로그인)
      → 결과 확인 (비로그인)
        → 찜/공유 시점에 카카오 로그인 유도
```

**동의 수집 구조** (입력 시작 직전, 별도 페이지 이동 없이 인라인):

| 구분 | 항목 | 보유 기간 |
|------|------|----------|
| **[필수]** 개인정보 수집·이용 | 직장 권역, 예산 범위 | 세션 종료 시 즉시 삭제 |
| **[선택]** 정밀 분석 추가 | 보유 현금, 상세 소득 | 세션 종료 시 즉시 삭제 |
| **[필수]** 위치정보 이용 | 직장 주소 → 좌표 변환 | 좌표 미저장 |

- 필수 미체크 시 CTA 비활성
- 선택 동의 없이도 기본 분석 가능

### 6-3. 안전 점수 표현 규칙 [source: P6 §4]

**✅ 사용 가능 표현:**
- 섹션 타이틀: "안전 편의시설 현황"
- 비교: "서울시 평균 대비", "수도권 상위 N%"
- 지표: CCTV 수, 가로등 밀집도, 경찰서 거리, 비상벨 수 (시설 수량/거리)
- 색상: 충분 = 녹색, 보통 = 주황, 부족 = **회색** (빨강 금지)

**❌ 사용 불가 표현:**
- "범죄율 높음", "위험 지역", "치안 열악", "치안 위험도"
- 원시 범죄 데이터 직접 노출
- 순위형 비하 (D등급, 하위 20%)
- 빨강 색상으로 "위험" 표현

### 6-4. 외부 링크 CTA 규칙 [source: P6 §5]

**CTA 버튼**: `외부 매물 보러가기 ↗ 네이버 부동산` + 하단 "외부 사이트로 이동합니다"

**클릭 시 모달 필수 요소**:
1. 이동 대상 사이트명 명시
2. 정보 제공 주체가 외부임 명시
3. 면책: "본 서비스는 정보 분석 플랫폼이며 부동산 중개 서비스가 아닙니다"
4. 확인/취소 양쪽 선택지

### 6-5. 면책 고지 위치 — 5접점 [source: P6 §6]

| # | 위치 | 톤 | 빈도 |
|---|------|-----|------|
| ① | 랜딩 하단 (모든 페이지 푸터) | 소프트 | 상시 |
| ② | 입력 폼 상단 배너 | 중립 | 1회 |
| ③ | 결과 상단 메타 태그 | 컨텍스트 내장형 (`[📊 공공데이터 기반] [참고용 정보]`) | 결과 확인 시 |
| ④ | 외부 링크 클릭 모달 | 명시적 | 매 클릭 |
| ⑤ | 이용약관 | 법률 문서 | 동의 시 |

## 7) 경쟁 대비 차별화 포인트 [source: P1, P2]

### 차별화 1: 커플 비동기 협업 보드

**기존 서비스에 없는 기능**: 한국 프롭테크 중 진정한 양방향 협업 보드를 제공하는 서비스 없음 [P2 §7]

- **구현 화면**: `/board/[id]` — 공유 찜 목록 + "둘 다 찜/한 명만 찜" 매칭 표시 + 블라인드 투표 + 인라인 코멘트·이모지 반응
- **근거**: Zillow 사용자 86%가 파트너와 공동 탐색, 73% 커플이 기준 합의를 스트레스로 인식 [P2 §1]. Redfin Shared Search 패턴으로 70%가 1개+ 찜 [P1 §1]

### 차별화 2: 맞벌이 이중 통근 동선 최적화

**기존 서비스에 없는 기능**: 두 직장 위치를 동시에 고려한 통근 분석은 기존 어떤 서비스도 미제공 [P2 §3]

- **구현 화면**: `/results` 지도 뷰 — 두 직장 마커 상시 표시 + 각 단지에서 양쪽 직장까지 통근시간 색상 오버레이, `/complex/[id]` — 직장1·2 소요시간 카드 + 요인 기여도
- **근거**: 맞벌이 커플이 타겟, Trulia 아이소크론 시각화 참고 [P1 §4], 호갱노노 통근 히트맵 확장 [P1 §4]

### 차별화 3: 종합 스코어링 시스템 (Niche식 등급 + 요인 분해)

**기존 서비스에 없는 기능**: 한국 부동산 5개 서비스 모두 종합 점수/레터 등급/레이더 차트 미사용 [P1 §3]

- **구현 화면**: `/complex/[id]` — 4축(통근·보육·안전·가성비) 0-100 점수 + 등급 배지(A+~D) + 요인 기여도 분해 + 백분위 비교, `/results` — 지도 마커에 점수 등급 색상 반영
- **근거**: Niche 레터 등급이 브랜드 정체성 자체, Walk Score 30,000+사이트 통합 [P1 §3, P2 §4]. 토스 신용점수 요인 공개 패턴 적용 [P2 §4]

# 신혼부부 맞춤형 주거 추천 서비스 — 리서치 종합 정리

> **문서 목적:** Gemini·GPT 두 건의 리서치를 Q&A Loop를 통해 의사결정한 뒤, 실제 개발에 바로 참고할 수 있도록 종합 정리한 문서
>
> **서비스 성격:** 개발 포트폴리오 + 본인 실사용 목적
>
> **기술 스택:** Backend(Java / Spring Boot / PostgreSQL) · Frontend(React / TypeScript / shadcn / TailwindCSS)

---

## 1. 서비스 핵심 정의

### 1.1 목적

기존 프롭테크 앱들은 매물 탐색·시세 확인에는 강하지만, **"내 예산 + 두 직장 통근 + 보육 + 치안"을 하나의 통합 랭킹으로 추천**해주는 서비스는 부재하다. 본 서비스는 신혼부부(본인 포함)가 **"어디 동네/단지가 우리 조건에 맞는지 발견"**하는 것을 핵심 가치로 한다.

### 1.2 핵심 흐름

```
[입력] 두 직장(프리셋) + 보유현금 + 가능 예산(전세금/매매가) + 월 상한 + 가중치
   ↓
[처리] 스코어링 모델로 단지별 점수 산출
   ↓
[출력] 지도 위 Top 10 단지 + 점수 breakdown + 추천 사유 + 외부 아웃링크
```

- 매물(호수) 데이터는 MVP에서 직접 제공하지 않음 → 네이버 부동산/호갱노노 아웃링크로 연결
- 대출 규제(LTV/DSR/특례)는 DB 기반 규칙 엔진으로 반영하여 예산 추정에 활용

### 1.3 타겟 사용자

- 1차: 본인(신혼부부, 맞벌이)
- 2차: 결혼 예정 또는 결혼 2년 이내의 수도권 맞벌이 신혼부부

## 2. 경쟁사 요약 — "뭐가 없는가"

두 문서(Gemini 8개사 분석 + GPT 스토어 기반 팩트 체크)를 종합하면, 주요 서비스별 **본 서비스 관점의 핵심 공백**은 다음과 같다:

| 서비스 | 포지셔닝 | 핵심 공백 (본 서비스 관점) |
|---|---|---|
| **네이버 부동산** | 매물 데이터 허브 | 개인화 추천 부재, 정보 과부하 |
| **직방** | 모바일 부동산 포털 | 2직장 D2D + 보육/치안 통합 추천 없음 |
| **호갱노노** | 지도 기반 커뮤니티 | 이중 통근 알고리즘 부재, 개인 재무 기반 필터 없음 |
| **KB부동산** | 금융 유틸리티 | 대출 계산기는 강하나 생활/안전 정보 미흡, UX 낙후 |
| **리치고** | 투자 분석 AI | 투자자 중심, 신혼부부 생활형 최적화 약함 |
| **아실** | 투자자 비교 도구 | 실거주 감성 데이터 부재 |
| **다방** | 1인가구 전월세 | 아파트 매매/전세 데이터 빈약 |

**종합 판단:** "재무입력 + 2직장 통근 + 보육 + 치안"을 단일 랭킹으로 통합한 신혼부부 특화 추천은 **명확한 시장 공백**이 있다.

## 4. 스코어링 모델

### 4.1 기본 방식: GPT식 선형 클리핑

MVP에서는 구현 속도와 디버깅 용이성을 위해 **선형 스코어링**을 채택한다. 추후 본인 사용 과정에서 점수 이상치를 발견하면 Gemini식 비선형(지수 페널티 등)으로 고도화한다.

Spring Boot에서 스코어링 로직을 **Strategy 패턴으로 분리**하여 선형/비선형 전환이 용이한 구조로 설계.

### 4.2 점수 산출 (0~100)

**Hard Filter (제외 조건):**
- `통근_max > 60분` → 제외
- `월고정비 > 월상한` → 제외

**Score = 100 × Σ(w_i × s_i)**

| 항목 | 산출식 | 비고 |
|---|---|---|
| 예산 (`s_budget`) | `clip(1 - (월고정비/월상한 - 1) / 0.3, 0, 1)` | 30% 초과부터 급격 페널티 |
| 통근 (`s_commute`) | `1 - clip(통근_max / 60, 0, 1)` | ODsay D2D 기반, 두 직장 중 긴 쪽 |
| 보육 (`s_childcare`) | `clip(1 - (도보근사거리_m / 800), 0, 1)` | 800m ≈ 10분 근사 |
| 치안 (`s_safety`) | `구별_범죄율_점수(0~1) × (1 + α×CCTV밀도 + β×경찰서근접도)` | 정규화 후 적용 |
| 학군 (`s_school`) | `통학구역_일치(1/0) 또는 거리_정규화` | SHP 기반 조회 |

### 4.3 가중치 프리셋 3안

| 프리셋 | 예산 | 통근 | 보육 | 치안 | 학군 |
|---|---:|---:|---:|---:|---:|
| **밸런스** | 0.35 | 0.35 | 0.15 | 0.10 | 0.05 |
| **통근 우선** | 0.30 | 0.45 | 0.10 | 0.10 | 0.05 |
| **육아 우선** | 0.20 | 0.30 | 0.25 | 0.10 | 0.15 |

- UX: "프리셋 3개 버튼" + Collapsible 세부 슬라이더(각 0.05~0.60, 합 자동 정규화)
- 이중 통근 균형(독박 통근 방지)은 **옵션 토글**로 제공: ON 시 `|통근A - 통근B| > 20분`이면 추가 페널티

### 4.4 월 고정비 추정

```
월고정비 = 관리비(K-apt) + 주거비
  - 전세: 전세대출 이자(추정)
  - 매매: 원리금 상환액(추정)
```

결과에 산출 근거 툴팁 제공 (데이터 기준일, 가정 사항 명시).

## 5. 대출 규제 반영 — DB 기반 규칙 엔진

### 5.1 설계 원칙

정책이 바뀌어도 **코드 배포 없이 DB에 row 추가**만으로 대응 가능한 구조.

### 5.2 테이블 설계

**`regulation_zone` — 규제지역 지정 이력**

| 컬럼 | 타입 | 설명 |
|---|---|---|
| zone_id | PK | |
| region_code | VARCHAR | 법정동 코드 |
| zone_type | ENUM | SPECULATIVE / ADJUSTMENT / NONE |
| effective_from | DATE | 적용 시작일 |
| effective_to | DATE (nullable) | 적용 종료일, null = 현행 |

**`ltv_policy` — LTV 한도 규칙**

| 컬럼 | 타입 | 설명 |
|---|---|---|
| policy_id | PK | |
| zone_type | ENUM | 규제지역 유형 |
| home_count | INT | 0=무주택, 1=1주택, 2=다주택 |
| price_min / price_max | BIGINT | 주택가격 구간 |
| ltv_rate | DECIMAL | 예: 0.50, 0.70 |
| effective_from / effective_to | DATE | 이력 관리 |

**`dsr_policy` — DSR 한도**

| 컬럼 | 타입 | 설명 |
|---|---|---|
| policy_id | PK | |
| borrower_type | ENUM | INDIVIDUAL / COUPLE |
| loan_amount_min / max | BIGINT | 대출금액 구간 |
| dsr_rate | DECIMAL | 예: 0.40 |
| effective_from / effective_to | DATE | |

**`special_loan` — 특례대출 (신생아 특례, 디딤돌 등)**

| 컬럼 | 타입 | 설명 |
|---|---|---|
| loan_id | PK | |
| loan_name | VARCHAR | 신생아특례_매매, 디딤돌 등 |
| income_max | BIGINT | 부부합산 연소득 상한 |
| asset_max | BIGINT | 순자산 상한 |
| marriage_years_max | INT | 혼인 기간 조건 |
| child_required | BOOLEAN | |
| interest_rate_min / max | DECIMAL | |
| ltv_override | DECIMAL | 특례 전용 LTV |
| loan_limit_max | BIGINT | 최대 대출한도 |
| effective_from / effective_to | DATE | |

### 5.3 처리 흐름 (Spring Boot)

```
사용자 입력 (연봉, 현금, 주택 수, 혼인 기간, 자녀 유무)
    │
    ▼
PolicyService.calculate(userInput, targetApartment)
    ├─ 1) regulation_zone에서 단지의 현행 규제지역 유형 조회
    ├─ 2) ltv_policy에서 조건 매칭 → 최대 LTV 결정
    ├─ 3) dsr_policy에서 DSR 한도 조회 → 연소득 기반 최대 상환액 계산
    ├─ 4) special_loan에서 적용 가능한 특례 필터링 → 금리/한도 우대 적용
    ├─ 5) min(LTV 기반 한도, DSR 기반 한도, 특례 한도) = 최종 대출 가능 추정액
    │
    ▼
  예산 = 보유현금 + 최종 대출 가능 추정액
```

### 5.4 안전장치

- 결과에 항상 **"기준일"** 표시: "이 추정은 2026년 2월 기준 규제 정보 기반입니다"
- 결과를 **"추정 범위"**로 표시: "대출 가능 추정: 약 3.2억 ~ 3.8억 (특례 적용 시)"
- 외부 계산기 아웃링크(KB 대출계산기, 주택금융공사 시뮬레이터) 함께 제공

## 9. 치안 점수 설계 (P0)

### 9.1 구성

```
치안점수 = 구별_범죄율_점수(0~1) + α × CCTV_밀도_500m(정규화) + β × 경찰서_지구대_근접도(정규화)
```

### 9.2 데이터 처리

- **구별 범죄율:** 경찰청 5대 범죄 통계 → 인구 10만 명당 발생률로 정규화 → 서울 평균 대비 상대 점수
- **CCTV 밀도:** 단지 중심 반경 500m 내 CCTV 개수 (PostGIS `ST_DWithin` 활용)
- **경찰서/지구대 근접도:** 단지에서 가장 가까운 경찰시설까지 거리 (PostGIS `ST_Distance`)

### 9.3 표시 방식

- 점수 자체를 "안전/위험"으로 단정하지 않음
- **"이 지역은 서울 평균 대비 5대 범죄율이 20% 낮으며, 반경 500m 내 CCTV 15대"** 식의 맥락 문장 제공
- 데이터 기준일·출처·한계 고지 포함
- 사용자 OFF 토글 제공 (비활성화 가능)

## 10. 기술 아키텍처 (요약)

### 10.1 백엔드 (Spring Boot + PostgreSQL)

- **PostGIS 확장:** 공간 쿼리(CCTV 반경, 경찰서 거리, 어린이집 반경, 통학구역 SHP)
- **배치 잡:** 공공데이터 API 수집 → ETL → DB 적재 (Spring Batch 또는 Scheduled Task)
- **통근 사전계산:** `commute_preset` 테이블 (업무지구 × 단지그룹 → 소요시간)
- **규칙 엔진:** `regulation_zone`, `ltv_policy`, `dsr_policy`, `special_loan` 테이블 기반 `PolicyService`
- **스코어링 API:** `POST /api/v1/recommend` → 입력 조건 기반 Top 10 단지 + 점수 반환

### 10.2 프론트엔드 (React + TypeScript + shadcn)

- **Kakao Map SDK:** 지도 렌더링, 커스텀 마커, 오버레이
- **shadcn/ui:** Card, Tabs, Select, Slider, Sheet, Table, Collapsible, Progress, Button
- **TailwindCSS:** 반응형 레이아웃 (사이드패널 + 지도 분할)
- **상태 관리:** 가중치 변경 시 프론트에서 스코어 재계산 (API 재호출 없이 즉시 반영)

## 11. 열린 질문 / 추가 리서치 필요 사항

Q&A Loop와 두 문서를 종합하면서 발견한, 아직 결정되지 않았거나 추가 확인이 필요한 사항들:

### 데이터 관련

1. **ODsay 유료 전환 시점:** 무료 1,000회/일로 프리셋 사전계산을 완료하려면 며칠이 걸리는가? 11개 업무지구 × 단지 ~7,000개 = ~77,000회. 단순 계산으로 약 77일. 격자 그룹핑(500개)으로 줄이면 ~5,500회 = 약 6일. → **격자 그룹핑 필수, 격자 크기 결정 필요**
2. **K-apt 관리비 API의 실제 커버리지:** 300세대 이상 의무관리 기준이면, 서울 아파트 중 몇 %가 커버되는가?
3. **초등 통학구역 SHP 처리:** PostGIS로 SHP 임포트 후 `ST_Contains`로 단지 좌표 매칭 — 성능 테스트 필요
4. **어린이집 API 호출 제한:** "기관 정책 따라 상이" → 실제 일일 한도 확인 필요

### 규제/정책 관련

5. **규제지역 지정 이력 데이터:** 법정동 코드 기반으로 현행 + 과거 규제지역 목록을 어디서 구할 수 있는가?
6. **특례대출 조건 변경 주기:** 신생아 특례/디딤돌 등의 조건이 얼마나 자주 바뀌는가? 운영 부담 추정 필요
7. **전세대출 이자 추정:** 전세자금대출 금리는 은행/상품별로 다른데, MVP에서 어떤 기준금리를 사용할 것인가? (예: COFIX 연동 평균)

### UX/제품 관련

8. **"추천 사유 문장" 템플릿:** 어떤 수준의 자연어 설명이 신뢰를 주는가? (예: "이 단지는 강남역 45분, 여의도역 30분 거리이며, 반경 500m 내 국공립 어린이집 2곳이 있습니다")
9. **MVP 지역 범위:** 서울 전체 + 성남(분당)/과천/하남 vs 서울만 vs 수도권 전체? → 데이터 정제 공수와 직결
10. **포트폴리오 시연 시나리오:** 어떤 입력 조건으로 데모하면 가장 인상적인가?

## 12. 수요 검증 임계치 (Go/No-Go)

### 12.1 Go 조건 (Step2 종료 시점)

아래 3개를 모두 만족하면 Go:

1. 인터뷰 20명 중 14명 이상이 현재 집 탐색 프로세스를 `4/5 이상 고통`으로 평가
2. 응답자 50% 이상이 `연봉/대출/직장 주소` 입력을 조건부라도 수용
3. 행동 KPI 충족:
   - 랜딩 실험 문의 클릭률 12% 이상, 또는
   - 컨시어지 리포트 문의 클릭률 20% 이상

### 12.2 No-Go / Pivot 조건

아래 중 하나라도 강하게 관측되면 피벗 검토:

1. 클릭률은 높은데 민감정보 입력 거부가 높음  
   -> `월상한 중심 간편 입력 모드`로 전환
2. 전세/매매 중 한 탭에서만 반응이 집중됨  
   -> 반응 높은 탭을 P0 핵심으로 축소
3. API 호출/데이터 결측으로 Top10 품질이 불안정함  
   -> 지역 범위 축소 + 사전계산 격자 최적화 우선

### 12.3 Step2 종료 체크리스트

- Problem/Solution fit이 인터뷰에서 반복 확인되었는가
- 입력 민감도(연봉/대출/직장 주소) 수용률이 절반 이상인가
- 문의 클릭률이 임계치 이상인가
- 단지 추천에 필요한 데이터 파이프라인이 안정 연결되었는가
- 대표 매물 아웃링크/문의 플로우가 실제로 동작하는가

1. **매물(호수) 데이터 미확보**
   - 리스크: 공공 실거래 데이터로 호수 단위 매물 대체 불가

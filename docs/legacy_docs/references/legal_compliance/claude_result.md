# 신혼부부 주거 추천 서비스 — 법적 규제 & 컴플라이언스 가이드

> **작성일**: 2026-02-13  
> **전제 조건**: 공인중개사 자격 미보유 / 영입 계획 없음 / 수익 모델 미정 (프리미엄 구독 + 금융상품 정보 제공 링크 검토 중)  
> **서비스 포지셔닝**: "부동산 정보 분석 플랫폼" (중개 플랫폼 아님)  
> **⚠️ 면책**: 본 문서는 법적 자문을 대체하지 않습니다. 서비스 출시 전 반드시 전문 법률 자문을 받으시기 바랍니다.

## Executive Summary

이 서비스는 **6개 법적 규제 영역**이 교차합니다. 가장 위험한 영역은 공인중개사법(중개행위 해당 여부)이며, 서비스 설계 방식에 따라 형사처벌까지 가능합니다. 아래 원칙을 준수하면 대부분의 리스크를 관리할 수 있습니다.

### 4대 설계 원칙

1. **"정보 제공 플랫폼"으로 포지셔닝** — 거래 알선·중개 기능을 절대 포함하지 않음
2. **입력값 비저장 아키텍처** — 금융 민감 정보(연봉·대출)를 서버에 저장하지 않음
3. **사용자 직접 입력(self-reported) 방식** — 마이데이터·금융사 API 연동 없음
4. **외부 매물 링크 방식** — 타 플랫폼 데이터 크롤링 없음

## 규제 영역 ① — 공인중개사법

### 위험도: 🔴🔴🔴 최상위

### 적용 법률
- 공인중개사법 제2조(중개의 정의), 제9조(개설등록), 제48조(벌칙)

### 핵심 쟁점
공인중개사 자격 없이 중개업 등록도 하지 않은 상태에서, 서비스가 객관적으로 "거래의 알선·중개를 위한 행위"로 인정되면 **3년 이하 징역 또는 3천만 원 이하 벌금** 대상입니다.

법원은 중개행위 해당 여부를 **행위자의 주관적 의사가 아니라 객관적·사회통념상 판단**합니다. "우리는 정보 제공만 합니다"라는 주장은 실질적 행위가 거래 알선이면 보호받지 못합니다.

또한 부동산 컨설팅을 명목으로 하더라도, 실질적으로 거래를 연결하면 중개행위로 판단됩니다. 해당 보수 약정은 무효이며 형사처벌 대상이 됩니다.

### CTA 설계: 안전 vs 위험 스펙트럼

| 방식 | 리스크 | 판단 근거 |
|------|--------|----------|
| ✅ 외부 매물 페이지 링크 (네이버 부동산 등) | 🟢 안전 | 단순 정보 연결, 거래 알선 아님 |
| ✅ 해당 지역 중개사무소 리스트 노출 (연락처 없이 상호·주소만) | 🟢 안전 | 공공정보 재가공, 거래 특정 안 됨 |
| ⚠️ 특정 매물에 대해 특정 중개사 전화번호 연결 | 🔴 위험 | 매물·중개사·사용자 3자를 연결 → 알선 해당 가능 |

### 핵심 판단: 신용정보법 적용 여부

| 시나리오 | 적용 법률 | 규제 수준 |
|---------|----------|----------|
| ✅ **사용자가 직접 입력** (연봉·대출 구간 선택) | 개인정보보호법만 적용 | 🟢 일반 |
| ⚠️ 마이데이터 연동 (은행 계좌·대출 자동 조회) | 신용정보법 + 본인신용정보관리업 등록 | 🔴 매우 높음 |
| ⚠️ 금융사 API 연동 (대출한도 실시간 조회) | 신용정보법 + 대출비교플랫폼 등록 | 🔴 매우 높음 |

**MVP는 반드시 "사용자 직접 입력" 방식만 사용**해야 합니다. 마이데이터 연동이나 금융사 API 연동은 신용정보법상 별도 등록이 필요하며, 자기자본 5억 원 이상 등 스타트업 단계에서 진입 장벽이 매우 높습니다.

### 입력값 비저장 아키텍처 (권고)

```
[사용자 입력] → [서버: 실시간 계산] → [결과 반환] → [입력값 즉시 삭제]
                                                    ↓
                                         결과만 저장 (점수, 추천 단지 리스트)
                                         금융 민감 입력값은 미저장
```

이 방식 채택 시 개인정보 보유 리스크가 대폭 감소합니다. 단, 재방문 시 재입력이 필요하므로 UX 트레이드오프가 있습니다.

### ⚠️ 주의: "로그에만 저장"도 동일한 리스크

DB INSERT를 하지 않더라도 **로그 파일에 개인정보가 남으면 법적으로 동일한 리스크**가 발생합니다.

개인정보보호법 제2조 제2호는 개인정보의 수집, 생성, 연계, 연동, 기록, **저장**, 보유, 가공, 편집, 검색, 출력, 정정, 복구, 이용, 제공, 공개, 파기 등 모든 행위를 "처리"로 정의합니다. 즉 DB든 파일 로그든 메모리 덤프든, **특정 개인을 식별할 수 있는 정보가 어딘가에 남아 있으면 전부 "저장·보유"에 해당**합니다.

#### 저장 방식별 법적 판단

| 저장 방식 | 개인정보보호법 적용 | 위치정보법 적용 | 실질 리스크 |
|----------|-----------------|--------------|-----------|
| DB INSERT | ✅ 적용 | ✅ 적용 | 기준 |
| 애플리케이션 로그 파일 | ✅ 적용 | ✅ 적용 | **동일** |
| 서버 접속 로그 (access log) | ✅ 적용 | ✅ 적용 | 동일 |
| 디버깅용 임시 로그 (24시간 후 삭제) | ✅ 적용 (보유기간 짧을 뿐) | ✅ 적용 | 약간 낮지만 여전히 적용 |
| 메모리에서만 처리 후 어디에도 미기록 | 🟢 저장·보유 아님 | 🟢 저장 아님 | ✅ 안전 |

핵심은 "서울시 강남구 테헤란로 123 + 사용자 세션 ID"가 로그에 남으면, 그 조합만으로 특정 개인의 근무지를 식별 가능하기 때문에 개인정보에 해당합니다.

#### 로그 저장 시 추가로 발생하는 의무

로그에 개인정보가 포함되면 DB와 똑같이 다음 의무가 따릅니다:

- **처리방침에 로그 보관 사실 명시** (보관 항목, 보유기간, 파기 방법)
- **접근통제** — 로그 파일에 대한 접근 권한을 최소 인원에게만 부여
- **암호화** — 고유식별정보·비밀번호 포함 시 암호화 저장 의무
- **보유기간 경과 후 파기** — 로그 로테이션으로 자동 삭제되더라도 정책 문서화 필요
- **위치정보법상 이용내역 보관** — 위치정보 수집·이용·제공 사실 확인자료를 6개월 이상 보관해야 하는 별도 의무도 있음

#### 진정한 "비저장" 구현을 위한 체크포인트

```
✅ 안전한 설계:

[사용자 입력: "서울시 강남구 테헤란로 123"]
    ↓
[서버 메모리: Geocoding API 호출 → 좌표 반환 → 통근시간 계산]
    ↓
[결과만 반환: "A 단지 통근시간 35분"]
    ↓
[입력 주소·좌표 → 메모리에서 해제, 로그 미기록]

⚠️ 실무에서 자주 놓치는 유출 경로:

1. 웹서버 access log — 쿼리 파라미터로 주소가 찍히지 않도록 POST 방식 사용
2. 에러 로그 — request body가 덤프되지 않도록 로깅 레벨·포맷 설정 확인
3. APM/모니터링 도구 — Sentry, Datadog 등이 에러 발생 시 request payload를 자동 수집
   → 반드시 마스킹(scrubbing) 처리 적용
4. 클라우드 로그 서비스 — AWS CloudWatch, GCP Logging 등에 자동 전송되는 로그 확인
5. 리버스 프록시 — Nginx/ALB access log에 request body가 기록되는지 확인
```

> **[권고]** "로그에만 남기니까 괜찮다"는 논리는 법적으로 통하지 않습니다. 비저장 원칙을 채택했다면 **DB뿐 아니라 모든 로그·모니터링·APM에서 개인정보가 완전히 배제되는 설계**가 필요합니다.

### 실행 조치

| # | 조치 | 시점 | 담당 |
|---|------|------|------|
| 2-1 | 개인정보 수집·이용 동의서 작성 (수집 목적, 항목, 보유기간, 필수/선택 분리) | 출시 전 | 법무 |
| 2-2 | 개인정보 처리방침 작성 및 홈페이지 게시 | 출시 전 | 법무/개발 |
| 2-3 | 입력값 비저장 아키텍처 구현 (연봉·대출 정보 서버 미저장) | 개발 시 | 개발 |
| 2-3a | 로그·모니터링 개인정보 배제 확인: access log POST 방식 전환, 에러 로그 request body 미덤프, APM(Sentry/Datadog) payload 마스킹, 클라우드 로그 서비스 확인 | 개발 시 | 개발/인프라 |
| 2-4 | 안전성 확보 조치: 전송 암호화(TLS 1.2+), DB 암호화(AES-256), 접근권한 관리, 접속기록 1년 보관 | 개발 시 | 개발/인프라 |
| 2-5 | 개인정보보호 책임자(CPO) 지정 | 출시 전 | 대표 |
| 2-6 | 제3자 제공 동의서 별도 작성 (금융사 제휴 도입 시) | 금융 제휴 시 | 법무 |

## 규제 영역 ③ — 위치정보법

### 위험도: 🟡 중간

### 적용 법률
- 위치정보의 보호 및 이용 등에 관한 법률 제9조(위치기반서비스사업 신고), 제16조(보호조치)

### 핵심 판단

| 시나리오 | 신고 필요? | 근거 |
|---------|----------|------|
| 사용자가 직장 주소를 텍스트로 입력 | 🟡 모호 | GPS 수집은 아니나, 주소+사용자 결합 시 위치 특정 가능 |
| 사용자 현재 위치(GPS) 기반 추천 | 🔴 필수 | 개인위치정보 직접 수집 |
| 아파트 단지 좌표 지도 표시 | 🟢 불필요 | 사물위치정보, 개인 아님 |

**권고**: 보수적으로 위치기반서비스사업 신고를 진행합니다. 신고제이므로 비용 없이 시간만 소요되며, 미신고 시 3년 이하 징역 또는 3천만 원 이하 벌금 리스크를 제거할 수 있습니다.

### MVP 설계 원칙

```
- 사용자 기기의 GPS/Wi-Fi 위치를 절대 수집하지 않음
- 직장 주소는 텍스트 입력 → 카카오맵 Geocoding API로 좌표 변환
- 변환된 좌표는 통근시간 계산에만 사용
```

# 1인 개발자를 위한 신혼부부 주거 추천 서비스 기획 방향

## 목적과 포지셔닝

- **목적** – 포트폴리오용으로 시장성 있는 서비스를 설계·구현하는 것이 목표이며, 실제 마케팅/상업적 운영은 계획하지 않습니다. 
- **포지셔닝** – 서비스는 부동산을 *중개*하지 않고 사용자의 재무·통근·생활 선호도를 기반으로 아파트 단지를 추천하는 **정보 분석 플랫폼**으로 자리매김해야 합니다. 경쟁 서비스들은 예산·통근·보육·치안 등 조건을 통합한 추천 기능을 제공하지 않아 명확한 공백이 존재하며【103981203960773†L11-L16】【103981203960773†L41-L53】, 이 부분을 포트폴리오에서 강조할 수 있습니다.

## 핵심 가치 제안

- **예산 역산 + 통근 최적화** – 사용자가 입력한 보유현금·연소득·대출잔액으로 가용 예산을 역산하고 두 직장의 대중교통 통근시간을 계산해 통근 기준을 만족하는 지역을 제시합니다.
- **보육·학군·치안 데이터 통합** – 어린이집 접근성, 학교 학업 성취도, 안전 인프라(범죄율과 CCTV·가로등 밀도 등)를 정량화하여 가중치별로 종합 점수를 산출합니다. 기존 플랫폼에서 보육·치안 정보가 추천 알고리즘에 포함되지 않은 점을 차별화 포인트로 삼습니다【103981203960773†L44-L53】.
- **Top‑N 단지 추천 및 설명 가능한 결과** – 최종 추천 결과는 상위 10개 단지명, 월 고정비 추정, 추천 사유(예: 통근 45분/보육시설 8곳/안전 인프라 우수 등)로 구성합니다.

## 포트폴리오용 MVP 범위

단일 개발자로서 8~12주 내에 완성할 수 있는 범위에 집중합니다. 다음 기능을 P0(필수)와 P1(추가)로 구분해 우선순위를 설정하세요.

### P0 – 필수 기능 (8주 내)

- **사용자 입력 폼** – 보유현금·합산 연봉·기존 대출액·직장1/2 주소·월 주거비 상한을 입력받습니다. 해당 정보는 메모리에서만 처리하고 서버에 저장하지 않는 **비저장 아키텍처**를 적용해야 합니다【543942488311355†L15-L21】【543942488311355†L94-L103】.
- **예산 산정 엔진** – 단순화된 LTV/DTI 모델을 사용해 가용 예산을 계산하고, 월 고정비(대출 원리금+관리비+예상 월세)를 산출합니다.
- **통근시간 계산** – ODsay API는 무료 호출 제한이 엄격하므로 포트폴리오에서는 서울 내 주요 업무지구와 아파트 단지 사이의 평균 통근시간을 미리 계산한 데이터 세트를 사용하거나, 직선거리 기반 근사치를 먼저 보여주고 상세 조회 시 실제 API를 호출하는 전략을 사용합니다.
- **단지 데이터 및 스코어링** – 국토교통부 공동주택 기본정보, 실거래가 API, 어린이집/학교/치안 등 공공 데이터를 내려받아 PostgreSQL/PostGIS에 적재합니다. 정규화된 점수(예산·통근·보육·학군·치안)를 가중치 프로필(균형형/통근중심형/예산중심형 등)으로 조합하여 **최종 점수**를 계산합니다.
- **결과 페이지** – 지도(카카오 맵 API 사용) 위에 추천 단지를 마커로 표시하고 상위 10개 단지의 카드(단지명, 위치, 월 고정비, 종합점수, 핵심 사유)를 보여줍니다. 클릭 시 외부 매물 페이지(네이버 부동산 등)로 이동하는 링크를 제공해 **중개 행위가 아님**을 분명히 합니다【543942488311355†L41-L49】.
- **기본 사용자 가중치 선택** – 예산 중심·통근 중심·균형형 중 하나를 선택하도록 하여 간단한 개인화 기능을 구현합니다.
- **기초 컴플라이언스** – 서비스 이용약관/개인정보처리방침 페이지를 작성해 “정보 제공 플랫폼이며 중개사가 아니다”, “모든 입력값은 계산 후 즉시 폐기된다”는 점을 명시합니다【543942488311355†L15-L21】. 입력값은 POST 방식으로만 전송하고, 로그/에러 로그에 개인정보가 남지 않도록 설정합니다.

### P1 – 추가 기능 (8~12주 차 또는 이후)

- **가중치 슬라이더** – 5개 요소(예산·통근·보육·학군·치안)에 대해 1–10 범위에서 세부 가중치를 조절할 수 있는 UI를 추가하여 맞춤형 추천을 강화합니다.
- **매물 샘플 표시** – 공공 실거래가를 기반으로 각 단지의 최근 실거래 평균과 예상 전세가를 표시하고, 대표 매물 1~2건(가상의 정보 혹은 제휴 중개사로부터 제공받은 샘플)을 노출합니다. 단, 연락처나 계약 중개는 제공하지 않습니다【543942488311355†L41-L49】.
- **카카오톡 공유** – 결과 요약을 카카오톡 공유 링크로 보내는 기능을 구현해 사용자가 배우자와 쉽게 공유할 수 있도록 합니다.
- **시각화 강화** – 점수 구성이 어떻게 되었는지 스택 바 차트나 레이더 차트로 시각화하여 사용자 신뢰를 높입니다.

## 데이터 및 기술 설계

- **데이터 수집** – 국토교통부 실거래가·공동주택 기본정보, 어린이집/학교 Open API, 생활안전지도(범죄율/CCTV/가로등) 등의 공공 데이터를 활용합니다. 각 데이터는 상업적 이용이 허용되며 출처를 표기해야 합니다【543942488311355†L82-L100】.
- **데이터 적재/전처리** – 초기에는 수도권(서울/경기/인천)의 주요 아파트 단지 몇 백 곳을 대상으로 데이터 세트를 구축합니다. Python 스크립트로 API를 호출해 CSV/JSON을 다운로드하고 PostGIS에 적재한 뒤, 반경 내 어린이집 수, 범죄 통계 등 보조 지표를 계산합니다.
- **백엔드** – Next.js API Routes 또는 Express.js를 사용하면 프런트와 동일한 코드베이스로 간단한 서버 기능을 구현할 수 있습니다. 입력 받은 데이터를 계산 엔진에 전달하고 결과를 반환하는 REST 엔드포인트를 제공합니다.
- **프런트엔드** – Next.js + Tailwind CSS로 모바일 우선의 반응형 UI를 작성합니다. 카카오 지도로 지도 인터랙션을 구현하고, shadcn/ui나 Material UI 같은 컴포넌트를 활용해 입력 폼과 결과 카드를 구성합니다.
- **캐싱/성능** – 통근시간과 같은 값비싼 API 호출 결과는 Redis나 메모리 캐시에 저장하고, 사전 계산된 결과를 우선 제시하는 전략을 사용합니다.
- **테스트/배포** – 개인 포트폴리오용이라면 Vercel(프론트)와 Supabase/AWS RDS를 사용해 손쉽게 배포할 수 있습니다. 데이터베이스에 실거래 데이터를 초기 적재할 때는 로컬에서 가공 후 RDS에 업로드합니다.

## 컴플라이언스 준수 및 리스크 관리

포트폴리오라고 해도 법적 리스크를 인지하고 설계에 반영해야 합니다.

- **중개 행위 회피** – 플랫폼을 “분석/안내” 서비스로 명확히 포지셔닝하고, 매물 클릭 시 외부 사이트로 이동하도록 설계합니다. 가격 협상, 계약 체결, 중개사 연결 등은 제공하지 않습니다【543942488311355†L41-L49】.
- **개인정보 최소 수집** – 신용정보법 적용을 피하기 위해 마이데이터 연동이나 대출금리 API를 사용하지 않습니다. 사용자가 직접 입력한 정보를 서버에 저장하지 않고 계산 후 파기하는 아키텍처를 유지합니다【543942488311355†L15-L21】.
- **위치정보법 준수** – GPS 기반 현재 위치 수집은 구현하지 않으며, 직장 주소 입력만으로 위치를 추정합니다. 보수적으로 위치기반서비스사업 신고를 검토하지만 포트폴리오 단계에서는 안내 문구로 갈음할 수 있습니다.
- **공공데이터 출처 표기** – 실거래가, 어린이집, 학교, 치안 등 모든 공공 데이터 옆에 출처를 표기하고 이용약관/하단에 통합 출처 영역을 두어 저작권 문제를 방지합니다【543942488311355†L82-L100】.
- **치안/안전 표현 주의** – “범죄율 상위 지역”처럼 부정적 단정을 피하고, CCTV·가로등·파출소 수 같은 **안전 인프라 지표** 형태로 표현합니다. 지역 낙인 논란을 방지하기 위해 점수 대신 별점이나 비교 범위를 제공합니다【543942488311355†L41-L49】.

이 기획은 **맞벌이 신혼부부에게 특화된 주거 추천을 제공하는 데이터 분석 서비스**를 1인 개발자가 현실적으로 구현할 수 있도록 단순화한 청사진입니다. 기존 플랫폼들이 제공하지 않는 통합 가치를 강조하고【103981203960773†L11-L16】【103981203960773†L44-L53】, 법적·윤리적 리스크를 최소화하는 설계를 선택했습니다. 포트폴리오 프로젝트라면 실제 사용자 데이터를 수집하지 않고도 공공 데이터를 활용하여 알맹이 있는 결과를 보여줄 수 있으므로, 위 계획을 바탕으로 차별화된 작업물을 완성하시길 바랍니다.

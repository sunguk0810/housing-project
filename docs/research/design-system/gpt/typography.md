# 한글 웹폰트 및 타이포그래피 시스템 대안 분석 리서치

## 연구 범위와 기준

본 리서치는 **신혼부부 대상 주거 분석(프롭테크) 서비스**의 “가격 정보 가독성”을 최우선 목표로, **모바일 퍼스트(390px 폭)** UI에서의 타이포그래피 시스템 대안을 비교·평가합니다. 기준일은 **2026-02-15 (KST)** 입니다.

분석은 (1) 폰트 배포/서빙 방식(CDN, 동적 서브셋), (2) 폰트 파일 크기/요청 구조, (3) 접근성/가독성(본문 최소 크기·줄간격), (4) 가격(숫자) 표기의 정렬/일관성(고정폭 숫자, 통화기호/구분기호) 관점으로 수행했습니다. 공공/범용 가이드 정합성은 **한국 디자인 시스템(KRDS)**의 타이포그래피 가이드를 기준으로 삼았습니다. citeturn54search0turn59view0

모바일 390px 화면의 “실 콘텐츠 폭”은 일반적인 좌우 패딩 16px 가정 시 **358px(=390-32)**이며, 한글 1글자 폭을 대략 1em으로 보면 “한 줄 글자 수(근사)”는 `358 / font-size(px)`로 계산할 수 있습니다(아래 ‘타입 스케일’ 섹션에서 수치 제시). 이 값은 **근사치**이며 실제는 폰트·문자(한글/숫자/기호) 구성에 따라 달라집니다.

## 한글 웹폰트 3종 정량 비교

비교 대상은 **Pretendard Variable(현행)**, **Wanted Sans**, **Noto Sans KR Variable**입니다. 요청하신 “11,172자 완성형 vs 2,350자 서브셋” 기준을 실제 배포 패키지 관점에서 분리해 보는 것이 중요합니다. 즉, “동일 서체”라도 **배포 패키지(Full vs Std/Subset)**에 따라 커버리지와 파일 크기가 달라집니다.

#### 커버리지(11,172 vs 2,350) 관점의 해석

“11,172자(현대 한글 완성형)”과 “KS X 1001(약 2,350자) 중심 서브셋”은 폰트 선택보다도 **배포 패키지 선택(풀셋 vs 스탠다드/서브셋)**에서 갈립니다.

- **Pretendard**는 **Full 패키지**(pretendard)와 **Std/Subset 패키지**(pretendard-std)가 분리 제공됩니다. 예를 들어 `pretendard`의 변수 WOFF2가 1.96MB인 반면 citeturn17view0, `pretendard-std`의 변수 WOFF2는 **284.84KB**로 훨씬 작아(즉, 커버리지 축소 가능성이 큼) “2,350자형 전략”에 가까운 배포가 가능합니다. citeturn10view0  
- **Noto Sans KR**는 “웹용 서브셋이 11,172 현대 한글 완성형을 커버”한다는 문구가 여러 배포 패키지(웹폰트 패키지)에서 반복됩니다. citeturn45search2turn45search5  
- **Noto Sans CJK KR(=CJK KR)**는 한글뿐 아니라 **한자(한글+Hanja) 등 광범위한 유니코드 블록**을 지원하는 초대형 커버리지이며(문자 44,806, 글리프 65,535 등), “11,172 vs 2,350”의 단순 비교를 넘어서는 범용 폰트입니다. citeturn39view0  

### 로딩 성능 해석(정량 지표 중심)

“초기 로드 시간”은 환경에 따라 달라지므로, 재현 가능한 비교를 위해 **초기 다운로드 바이트**와 **요청 구조(동적 서브셋 조각 수)**를 핵심 지표로 두는 것이 안전합니다.

- **Pretendard Variable 단일 파일(1.96MB)**을 “첫 화면에서 즉시 필요”하게 만들면(프리로드/FOIT 최소화 목적) 네트워크에 따라 체감이 커질 수 있습니다. citeturn17view0  
- 대신 **Pretendard Variable Dynamic Subset**은 변수 폰트를 **92개 조각**으로 나누어 필요한 글리프만 내려받게 하는 구조이며, CSS도 별도로 제공합니다. citeturn17view0turn16view0  
- **Noto Sans KR Variable**은 실무에서 “단일 파일”보다 **unicode-range 기반 다중 조각 다운로드**가 일반적이며, Fontsource도 이 접근을 강조합니다. citeturn63search10turn32view0  
- **Wanted Sans**도 “split(동적 서브셋) CSS” 제공이 공식 문서에서 확인됩니다. citeturn22view0turn70search0  

결론적으로, “모바일에서 가격 정보가 핵심”인 제품 UI에서는 **(A) Pretendard처럼 ‘동적 서브셋이 확실하고 운영 레퍼런스가 많은 폰트’**가 가장 예측 가능하며, **(B) Noto 계열은 커버리지가 강하지만 패키징/용량이 커지기 쉬워 ‘서브셋/요청 구조 설계’가 사실상 필수**입니다. citeturn63search2turn39view0turn32view0

## 타입 스케일 최적화

### KRDS 접근성·가독성 요구조건 재정리

KRDS 타이포그래피 가이드는 (1) **본문 최소 16px**, (2) **줄간격 150% 이상**, (3) 상대 단위(em/%/rem) 활용 등을 명시합니다. 또한 Pretendard GOV는 “체감 크기가 작아” 기본 크기를 17px로 쓰는 예시도 제시합니다. citeturn54search0turn59view0

따라서 현재 시스템의 **Body 16px / line-height 1.6(160%)**는 “KRDS 최소 기준”을 충족하는 쪽이며(16px 이상, 150% 이상), 확장 시에도 이 하한을 유지하는 방향이 합리적입니다. citeturn54search0

### 모듈러 스케일(비율)별 후보군과 모바일 390px 여유

요청하신 비율(1.125 / 1.2 / 1.25)을 “Base 16px(본문)” 기준으로 전개하면 대략 아래와 같습니다(소수점은 반올림/토큰화 과정에서 조정 가능).

| 비율 | Base 16 기준 상향 스케일(근사) | 성격 |
|---|---|---|
| 1.125 (Major Second) | 16 → 18 → 20.25 → 22.78 → 25.63 → 28.83 | 촘촘(본문·리스트 중심) |
| 1.2 (Minor Third) | 16 → 19.2 → 23.04 → 27.65 → 33.18 → 39.81 | 균형형(제품 UI에 무난) |
| 1.25 (Major Third) | 16 → 20 → 25 → 31.25 → 39.06 | 강한 대비(히어로/마케팅) |

모바일 390px(콘텐츠 폭 358px)에서의 “한 줄 글자 수(근사)”는 아래와 같습니다.

| 16 | ~22.4자 | 본문(하한선으로 적절) |

### 6단계 유지 vs 7–8단계 확장에 대한 제안

현재 6단계(12/14/16/17–18/20/24)는 “본문 중심 제품 UI”에는 충분하지만, **가격(핵심 수치)이 제품 가치 제안의 중심**이라면 “숫자 전용 강조 레벨”이 따로 있는 편이 운영이 쉽습니다.

KRDS의 헤딩 예시에는 모바일 기준 24/22/19/17/15 등의 계층뿐 아니라, 큰 헤딩(PC 40, 모바일 28 등)도 제시되어 있어 “상위 디스플레이 레벨을 별도 토큰으로 두는 방식”은 공공 가이드와도 충돌하지 않습니다. citeturn54search0

권장안은 아래와 같습니다.

- **앱/제품 UI 기본: 7단계**  
  - 기존 6단계를 정리해 **Subtitle을 18px로 고정**(17–18 범위 제거)하고,  
  - 최상단에 **Display 32px(가격·결과 점수 전용)**을 추가  
- **랜딩/마케팅 화면 한정: 8단계(옵션)**  
  - Hero 40px은 “한글 문장형”보다 “숫자/짧은 슬로건”에 적합하므로, 앱 내부의 정보 화면에는 기본 포함하지 않고 **마케팅/온보딩 일부 화면에서만** 사용

| 레벨 | 제안 px | 용도(핵심) |

## 가격 표시 전용 폰트 전략

가격 표시는 “정렬(자리수)·스캔 속도·단위 혼합(₩/쉼표/억·만원)” 문제로, 일반 본문 타이포보다 **미세한 일관성**이 더 중요합니다. 특히 목록(리스트)에서 여러 가격을 세로로 비교하는 UI라면, 숫자의 폭이 들쭉날쭉하면 스캔이 느려집니다.

### 한국 원화(₩), 쉼표 구분, “억/만원” 혼합 가독성 제안

실무적으로는 다음 3가지를 “가격 토큰”에 명시하는 편이 운영이 쉽습니다.

1) **정렬 규칙(리스트·표)**: 가격이 여러 줄로 반복되는 UI(매물 리스트, 비교표)에서는 숫자 폭을 고정폭으로 두는 편이 스캔이 빠릅니다(가능하면 tabular figures). Wanted Sans 제작기에서 고정폭 숫자를 직접 다루는 맥락이 나오며, 이는 가격 UI와 결이 같습니다. citeturn53view0  

2) **통화 기호(₩)와 쉼표(,)의 “붙는 느낌” 최소화**: Pretendard 관련 글에서 “₩ 등의 기호를 일상적으로 쓰이는 모습으로 맞추고 숫자 폭도 조정”했다고 언급되는데, 이런 의도는 가격 UI 안정성에 직접 기여합니다. citeturn71view0  

3) **단위 혼합(예: ‘3억 5,000만’)**: 숫자-단위-숫자 구조가 반복되면, 단위를 “숫자보다 한 단계 낮은 크기/무게”로 주어 시각적 소음을 줄이고, 숫자 덩어리의 비교가 더 빨라지는 경우가 많습니다. 이때도 줄간격 150% 이상을 기본으로 두면 가독성 리스크가 낮아집니다. citeturn54search0

### 이번 리서치의 정량 공백과 보완 방법

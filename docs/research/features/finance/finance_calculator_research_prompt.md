# 금융 계산기 기능 종합 리서치 프롬프트

## 역할

당신은 PropTech 서비스의 시니어 Product Analyst입니다.  
아래 서비스 맥락과 제약 조건을 숙지한 뒤, 금융 계산기 기능 도입에 대한 종합 리서치를 수행하세요.

---

## 서비스 맥락

- **서비스명**: 집콕신혼 — 수도권 맞벌이 신혼부부 대상 주거 분석 정보 플랫폼
- **법적 포지션**: `정보 제공/분석 도구` (중개/알선/금융자문 금지)
- **타겟**: 서울 직장 기준 맞벌이 신혼부부 (대부분 생애최초 주택구매자)
- **핵심 플로우**: 조건 입력(5스텝) → Top10 단지 분석 결과 → 외부 매물 아웃링크
- **기술 스택**: Next.js + Tailwind CSS v4 + Recharts + Supabase
- **수익 모델(P0)**: 무료 정보 제공 (거래성사형 CPA 금지)
- **데이터 원칙**: 개인정보 비저장(세션 단위 파기), 크롤링 금지, 공공/API/제휴만 허용

### 기존 온보딩 5스텝 구조

| Step | 제목 | 입력 항목 |
|------|------|---------|
| 1 | 우리 부부는요 | 주거 형태(매매/전세/월세) + 자녀 계획 |
| 2 | 출퇴근 정보 | 직장1 + 직장2 주소 |
| 3 | 소득과 자산 | 합산 연봉 + 보유 현금 |
| 4 | 부채와 예산 | 기존 대출 + 월 주거비 |
| 5 | 분석 중… | 결과 전환 애니메이션 |

### 금융 계산기 배치 계획

계산기를 **두 곳**에 배치하려고 합니다:

1. **온보딩 Step 3-4 통합**: 예산/부채 입력 과정에서 자연스럽게 LTV/DTI/DSR 참고값을 제공
2. **별도 독립 페이지 (`/calculator`)**: 네이버 금융 계산기처럼 범용 도구로 제공

### 검토 대상 계산기 유형 (네이버 금융 계산기 참조)

| 계산기 | 설명 | 핵심 입력값 |
|--------|------|-----------|
| 대출 계산기 | 월 상환액, 총 상환금액, 이자 총액 산출 | 대출금액, 기간, 금리, 상환방법(원리금균등/원금균등/만기일시) |
| LTV 계산기 | 담보인정비율 기반 대출 가능 금액 산출 | 대출목적, 보유 주택 수(생애최초/서민실수요/무주택/1주택/2주택이상), 지역(시/도·시/군/구), 주택 시세 |
| DTI 계산기 | 총부채상환비율 산출 | 연소득, 주택담보 대출금액/기간/금리/상환방법, 보유대출 연이자 상환액 |
| DSR 계산기 | 총부채원리금상환비율 산출 | 연소득, 주택담보 대출금액/기간/금리/상환방법, 보유대출 연이자+연원금 상환액, 스트레스 DSR 3단계(변동형/혼합형/주기형) |
| 중도상환수수료 | 중도상환 시 수수료 산출 | (상세 미조사) |

## 법적 제약 (필수 준수)

1. **서비스 포지션**: `정보 제공/분석 도구`로 고정. 금융 상품 추천/비교중개/모집 행위 금지.
2. **금지 표현**: "대출 가능 보장", "대출 승인 가능", "예산 적정", "투자 수익률 보장" 등 단정형 표현 금지.
3. **허용 표현**: "예상 대출 범위 (참고용)", "참고용 시뮬레이션 결과", "참고 범위 이내".
4. **면책 필수**: 모든 계산 결과에 "승인/가능 보장 아님" 면책 문구 고정 표시.
5. **개인정보 비저장**: 계산기 입력값도 세션 단위 파기, DB/로그/APM 비저장 원칙 동일 적용.
6. **출처 표기**: 계산 공식/기준의 출처(한국은행, 금융위원회 등)와 기준일 표시 필수.

## 리서치 요청 (5개 축)

### 축 1: 기능 범위 및 우선순위 결정

다음을 분석하세요:

1. 위 5개 계산기(대출/LTV/DTI/DSR/중도상환수수료) 중 **신혼부부 타겟에 실질적 가치가 높은 순서**대로 우선순위를 매기세요.
2. 각 계산기에 대해 **MVP(P0)에 포함할지 / P1으로 미룰지 / 제외할지** 판정하고 근거를 제시하세요.
3. 판정 기준으로 다음 4가지를 사용하세요:
    - 타겟 사용자(생애최초 주택구매 신혼부부)의 의사결정 기여도
    - 온보딩 플로우(Step 3-4)와의 자연스러운 연결 가능성
    - 독립 페이지(`/calculator`)에서의 SEO/유입 가치
    - 구현 복잡도 대비 사용자 가치

### 축 2: 법적 리스크 및 금융 규제 분석

다음을 분석하세요:

1. 각 계산기 유형별로 **금융자문/금융중개 오인 리스크 수준**(High/Medium/Low)을 판정하세요.
2. 특히 아래 시나리오의 법적 해석을 검토하세요:
    - 사용자가 입력한 조건으로 LTV/DSR을 자동 계산해서 "대출 가능 금액"을 표시하는 행위
    - 계산 결과를 기반으로 Top10 단지 결과의 예산 적합도 점수에 반영하는 행위
    - 특정 은행/금융상품 금리를 참조하여 계산하는 행위 vs 사용자 직접 입력 금리로만 계산하는 행위
3. **반드시 포함해야 할 면책/고지 요소**를 위치별로 정리하세요.
4. 계산기 결과 화면에서 사용 가능한 표현 / 사용 불가 표현 예시를 각 5개 이상 제시하세요.

### 축 3: UI/UX 패턴 조사 (경쟁사/유사 서비스)

다음을 조사하세요:

1. **한국 서비스** 중 금융 계산기를 잘 구현한 사례 3개 이상을 분석하세요.
    - 참고 후보: 네이버 금융 계산기, 카카오뱅크, 뱅크샐러드, KB부동산, 토스, 하나은행 등
    - 각 사례에서: 입력 UI 패턴, 결과 표시 방식, 면책 처리 방식, 용어 설명 토글 유무
2. **온보딩 플로우에 계산기를 자연스럽게 통합한 사례**가 있는지 조사하세요.
    - 부동산/금융 서비스에서 입력 과정 중 실시간 계산 피드백을 제공하는 패턴
3. 독립 계산기 페이지의 **SEO 효과를 극대화한 사례**를 분석하세요.
    - 금융 계산기가 검색 유입 퍼널 역할을 하는 패턴
4. **신혼부부/생애최초 특화 계산기**가 존재하는지 조사하세요.
    - 생애최초 우대 LTV, 신혼부부 특별공급 자격 확인 등

### 축 4: 기술 구현 방식

다음을 분석하세요:

1. 각 계산기의 **계산 공식**을 정리하세요:
    - 대출 상환: 원리금균등/원금균등/만기일시 공식
    - LTV: 지역별/보유주택수별/대출목적별 LTV 비율 기준표 (2025~2026년 최신)
    - DTI: 산출 공식 + 규제 비율 기준
    - DSR: 산출 공식 + 스트레스 DSR 3단계 가산금리 기준 (2025~2026년 최신)
2. **클라이언트 사이드 계산 vs 서버 사이드 계산**의 장단점을 분석하세요.
    - 개인정보 비저장 원칙과의 정합성 관점 포함
3. LTV/DSR 기준표의 **데이터 소스 및 갱신 주기**를 정리하세요.
    - 공공데이터/금융위원회/한국은행 등에서 어떤 형태로 제공되는지
    - 규제 변경 시 업데이트 프로세스
4. 계산기에 필요한 **지역 데이터**(시/도, 시/군/구, 투기과열/조정대상 지역 분류)의 데이터 소스를 파악하세요.

### 축 5: 서비스 통합 설계 방향

다음을 제안하세요:

1. **온보딩 Step 3-4 통합 시나리오**: 사용자가 소득/자산/부채를 입력하는 과정에서 계산기가 어떤 시점에, 어떤 형태로 개입하면 자연스러운지 구체적 플로우를 제안하세요.
    - 입력 도중 실시간 피드백 vs 입력 완료 후 요약 제공 vs 선택적 "더 알아보기" 확장
    - 기존 5스텝 위저드의 완료 시간(~2분 30초)을 크게 늘리지 않는 방안
2. **독립 페이지(`/calculator`) 설계 방향**: 네이버 금융 계산기 수준의 독립 도구로서 필요한 기능과 구조를 제안하세요.
    - 탭 구조(대출/LTV/DTI/DSR 등)
    - 용어 설명 토글(네이버 참조)
    - 결과 → 본 서비스 분석 결과로의 자연스러운 연결(CTA) 설계
3. **두 배치의 기능 차이**: 온보딩 통합 버전과 독립 페이지 버전의 기능 범위 차이를 정리하세요.
    - 온보딩: 간소화된 핵심 계산만 vs 독립: 풀 기능
